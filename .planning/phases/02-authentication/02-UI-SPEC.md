## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding |
| sm | 8px | Input internal vertical padding, label-to-input gap |
| md | 16px | Input horizontal padding, card internal padding |
| lg | 24px | Form field vertical gap, card-to-card gap |
| xl | 32px | Section vertical gap inside page |
| 2xl | 48px | Logo/title block bottom margin |
| 3xl | 64px | Not used in auth pages |

Exceptions: Touch target minimum 44px **height** for all interactive elements (login/register submit buttons). This is a height constraint (`min-h-[44px]`), not a gap or padding token — do not use 44px as a spacing value anywhere in the layout. Achieved via `py-3 px-4` (12px + 4px = 16px vertical + text ~20px = ~36px) — bump to explicit `min-h-[44px]` on the submit button to meet the minimum.

Source: 8-point scale derived from existing `page.tsx` form patterns (`px-4 py-3`, `gap-2`, `mb-10`, `p-8`).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label | 14px (`text-sm`) | 600 (semibold) | 1.4 |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 |
| Display | 28px (`text-2xl`) | 600 (semibold) | 1.1 |

Notes:
- Two weights only: 400 (regular) and 600 (semibold). The size difference between Heading (20px) and Display (28px) provides sufficient visual hierarchy without a third weight.
- Page heading "Zaloguj się" / "Zarejestruj się": Heading role (20px semibold) — matches existing `h1 text-xl font-semibold text-buddy-text` from `page.tsx`.
- Subtitle line under heading: Body role (14px regular `text-buddy-text-muted`) — matches `p text-sm text-buddy-text-muted` from `page.tsx`.
- Input placeholder text: Body role, `text-buddy-text-faint` — matches existing input pattern.
- Error message text: Body role (14px regular), `text-buddy-error` (`#c85a3a`).
- Do not introduce new font sizes. These four roles cover all auth page needs.

Source: `page.tsx` typography patterns; `tailwind.config.ts` fontFamily tokens.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#141210` (`buddy-base`) | Page background, full-screen auth surface |
| Secondary (30%) | `#1e1a16` (`buddy-elevated`) | Input backgrounds, form card if used |
| Accent (10%) | `#c8902a` (`buddy-gold`) | Submit button fill, focus ring on inputs, logo mark gradient start |
| Destructive | `#c85a3a` (`buddy-error`) | Inline error messages under fields only |

Accent reserved for: submit button background, active input focus border (`focus:border-buddy-gold`), logo gradient mark. Not used on: link text, labels, decorative elements.

Border colors:
- Default input border: `#2a2520` (`buddy-border`)
- Hover/focus input border: `#c8902a` (`buddy-gold`) — transition 150ms
- Error state input border: `#c85a3a` (`buddy-error`)

Text colors:
- Primary text: `#e8dcc8` (`buddy-text`)
- Muted text (subtitle, hint): `#8a7a68` (`buddy-text-muted`)
- Placeholder: `#5a4e42` (`buddy-text-faint`)
- Error text: `#c85a3a` (`buddy-error`)

Source: `tailwind.config.ts` full buddy color palette; `page.tsx` form input pattern.

## Component Inventory

These are the visual components Phase 2 introduces or requires:

### 1. Auth Page Shell

Full-screen centered layout. No nav bar, no TopBar (unauthenticated state has no project context). The submit button filled with `buddy-gold` is the primary visual focal point on the auth page; the logo mark is the secondary anchor.

```
min-h-screen flex items-center justify-center bg-buddy-base p-8
  └── w-full max-w-[420px] px-2
       ├── Logo mark (44×44 rounded-2xl, gold gradient)
       ├── Heading (h1 text-xl font-semibold text-buddy-text)
       ├── Subtitle (p text-sm text-buddy-text-muted)
       └── Auth form card
```

Matches the existing `page.tsx` centered layout exactly — no new shell pattern needed.

### 2. Auth Form Card

No explicit card border by default (borderless form on dark base). If visual separation is desired, use `bg-buddy-elevated rounded-2xl p-8`.

Fields: Email, Password, (Register only: Password confirm — NOT required by AUTH-02, omit to keep scope minimal).

### 3. Form Input

Exact pattern from `page.tsx`:
```
bg-buddy-elevated border border-buddy-border rounded-xl px-4 py-3
text-sm text-buddy-text placeholder:text-buddy-text-faint
focus:outline-none focus:border-buddy-gold transition-colors duration-150
```

Error state variant:
```
border-buddy-error focus:border-buddy-error
```

### 4. Submit Button (Primary)

Exact pattern from `page.tsx` create button:
```
w-full px-4 py-3 min-h-[44px]
bg-buddy-gold rounded-xl text-sm font-medium text-buddy-surface
hover:bg-buddy-gold-light
disabled:opacity-40 disabled:cursor-not-allowed
transition-all duration-150
```

Loading state: button text changes to spinner indicator (e.g. "..." or inline SVG spinner). Do not disable cursor during loading — user may want to see the spinner. Keep `disabled` only when form is truly invalid.

### 5. Inline Field Error

Below the field that caused the error, not in a global banner:
```
mt-1 text-xs text-buddy-error
```

### 6. Page-Level Error (API errors)

For 401/409/422 responses from the backend, shown above the submit button:
```
mt-2 px-3 py-2 rounded-lg bg-buddy-error/10 border border-buddy-error/30
text-sm text-buddy-error
```

### 7. Auth Navigation Link

"Nie masz konta? Zarejestruj się" / "Masz już konto? Zaloguj się" — plain text link:
```
text-sm text-buddy-text-muted
  span.cursor-pointer.text-buddy-gold.hover:text-buddy-gold-light.underline-offset-2.hover:underline
```

## Interaction States

| State | Visual Behavior |
|-------|----------------|
| Input: default | Border `buddy-border`, background `buddy-elevated` |
| Input: focus | Border `buddy-gold`, no outline (CSS outline: none), transition 150ms |
| Input: error | Border `buddy-error`; error message appears below |
| Input: disabled | `opacity-40 cursor-not-allowed` |
| Button: default | Background `buddy-gold`, text `buddy-surface` |
| Button: hover | Background `buddy-gold-light`, transition-all 150ms |
| Button: loading | Text replaced with "..." or spinner; pointer-events-none; do not grey out |
| Button: disabled (invalid form) | `opacity-40 cursor-not-allowed` |
| Page: error (API) | Error box fades in above submit; input borders stay unchanged |

Auth transition (success): `router.push("/")` — no animation needed, Next.js page transition handles it.

Source: D-10 from CONTEXT.md; RESEARCH.md Pattern 8.

## Accessibility

- All form inputs have explicit `<label>` elements with `htmlFor` matching input `id`.
- Error messages use `role="alert"` so screen readers announce them on appearance.
- Submit button must not be the only focusable element — Tab order: email → password → submit → nav link.
- Password input uses `type="password"` (no show/hide toggle required for Phase 2).
- No color-only error signaling — error border is paired with error text below the field.
