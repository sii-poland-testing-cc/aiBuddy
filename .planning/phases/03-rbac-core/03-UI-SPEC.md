## Copywriting Contract

Phase 3 introduces no new frontend screens. The copywriting contract governs HTTP error responses that the existing `ErrorBanner` component must display when backend returns 401 or 403, and the response body of the new bootstrap endpoint.

### HTTP Error Copy (backend response bodies)

These strings are returned in JSON `{"detail": "..."}` from FastAPI and surface in the frontend via existing error handling:

| HTTP Status | Trigger | `detail` string |
|-------------|---------|-----------------|
| 401 Unauthorized | Missing or invalid JWT cookie | `Nieautoryzowany dostęp. Zaloguj się, aby kontynuować.` |
| 403 Forbidden — no role | User authenticated but has no role on the target resource | `Brak uprawnień do tego zasobu.` |
| 403 Forbidden — IDOR | Authenticated user requesting another user's resource | `Brak uprawnień do tego zasobu.` |
| 403 Forbidden — SSE | Permission denied on SSE endpoint before stream starts | `Brak uprawnień do uruchomienia tej operacji.` |
| 409 Conflict — bootstrap | Bootstrap called when users table is not empty | `Konto superadmina już istnieje.` |

Note: 401 vs 403 use different `detail` strings intentionally — 401 tells the user to log in; 403 tells them they lack permission. Both map to the same visual error container in the existing UI (`bg-buddy-error/10 border border-buddy-error/30 text-buddy-error`).

### Frontend 401 Redirect Behavior

The existing `frontend/middleware.ts` already handles the 401 scenario at the Next.js edge: any page request without an `access_token` cookie redirects to `/login`. API fetch 401 responses (from already-expired tokens mid-session) are not yet handled with client-side redirect — this is an acceptable gap for Phase 3; the next page load will trigger middleware redirect.

### Bootstrap Endpoint Response Copy

`POST /api/auth/bootstrap`:

| Scenario | HTTP Status | Response body |
|----------|-------------|---------------|
| Success — first superadmin created | 201 | `{"user_id": "<uuid>", "email": "<email>", "message": "Konto superadmina zostało utworzone."}` |
| Failure — users table not empty | 409 | `{"detail": "Konto superadmina już istnieje."}` |

### Empty State Copy

Not applicable — Phase 3 introduces no new data-displaying UI.

### Destructive Action Confirmation

Not applicable — Phase 3 has no destructive frontend actions. The `DELETE` routes being guarded do not receive new confirmation UI in this phase.

## Interaction States (Existing Components, New Backend Behavior)

The following interaction state changes occur without UI code changes — they result from the backend now enforcing guards that previously did not exist:

| State | Trigger | Visual Result | Component |
|-------|---------|---------------|-----------|
| Mid-session 401 | JWT expires during an active session | Next API call fails silently; next page navigation redirects to `/login` via middleware | `middleware.ts` |
| 403 on project load | User navigated directly to `/project/[id]` for a resource they have no role on | Fetch returns 403; existing error propagation surfaces the error text | Project page hooks |
| 403 on SSE start | User triggers audit/context-build/requirements-extract/mapping without permission | SSE connection receives 403 before any stream data; existing SSE hooks handle non-200 as error | `useAIBuddyChat.ts`, `useContextBuilder.ts`, `useRequirements.ts`, `useMapping.ts` |

All three states resolve using existing error display patterns — no new UI state machines needed.
