**models.py -- Add two columns and two relationships to the Project class (per D-08):**

Add these imports at the top (extend existing import line): `Optional` is already imported from typing.

Add these columns to the `Project` class body, AFTER the `settings` column and BEFORE the `files` relationship:

```python
# -- Hierarchy FKs (Phase 1) ------------------------------------------------
organization_id: Mapped[Optional[str]] = mapped_column(
    String,
    ForeignKey("organizations.id", ondelete="RESTRICT"),
    nullable=True,
    index=True,
)
workspace_id: Mapped[Optional[str]] = mapped_column(
    String,
    ForeignKey("workspaces.id", ondelete="SET NULL"),
    nullable=True,
    index=True,
)
```

Add these relationships AFTER the `snapshots` relationship:

```python
# -- Hierarchy relationships ------------------------------------------------
organization: Mapped[Optional["Organization"]] = relationship(
    "Organization", back_populates="projects"
)
workspace: Mapped[Optional["Workspace"]] = relationship(
    "Workspace", back_populates="projects"
)
```

Note: Use string forward references `"Organization"` and `"Workspace"` to avoid circular imports (hierarchy_models.py imports Base from models.py). SQLAlchemy resolves these strings at mapper configuration time after both modules are imported by engine.py.

`organization_id` is intentionally `nullable=True` in the ORM (diverging from D-08's "NOT NULL" requirement per Research recommendation). Rationale: SQLite prohibits ADD COLUMN NOT NULL without DEFAULT; the migration adds the column as nullable then fills via UPDATE. Using `nullable=True` in both ORM and migration avoids `alembic check` drift (Research Pitfall 2). The FK `ondelete="RESTRICT"` constraint provides runtime integrity — no project can reference a non-existent org. Phase 2 can tighten to NOT NULL if needed after PostgreSQL migration.
